# スタディペイ MVP開発計画

## 1. MVPの目的

スタディペイのMVPでは、課金・高度な履歴・詳細な写真保存期限管理よりも先に、親子の中心体験を最短で検証する。

中心に置く体験は以下とする。

1. 保護者が登録する。
2. 保護者が子どもを追加する。
3. 子どもが学習成果を申請する。
4. 保護者が内容と写真を確認する。
5. 保護者が承認・差し戻し・却下する。
6. 承認時にポイントが付与される。
7. 子どもがポイント残高を確認する。

この流れにより、「学習成果を親子で確認し、ポイントとして見える化する体験」が成立するかを検証する。

## 2. MVPで最初に作る範囲

### 含める

- 保護者サインアップ
- 保護者ログイン・ログアウト
- 保護者ホーム
- 子ども追加・一覧・詳細
- 子どもログインIDと初期パスワード発行
- 子どもログイン
- 子どもホーム
- 科目の初期設定
- シンプルなポイントルール設定
- テスト申請
- 成績申請

- 申請履歴

- 承認時のポイント付与
- ポイント残高表示
- アプリ内通知の最低限表示
- キャラクター画像は使わず、ロゴ・アイコン・文言で親しみやすさを補う

### 後回しにする

- 本番課金
- 14日間トライアルの厳密な利用停止制御

- 退会

## 3. MVPのPhase構成

### Phase 1: LP・認証・スマホ土台

目的: 未ログインから保護者ホームまで到達できる状態を作る。

実装タスク:

- LPを作成する。
- LPに「14日間無料で始める」「ログイン」を配置する。
- キャラクター画像はファーストビューに配置しない。
- 保護者サインアップ画面を作成する。
- 保護者ログイン画面を作成する。
- ログアウトを実装する。
- ログイン後の保護者ホームを作成する。
- スマホファーストの共通レイアウトを作成する。
- 下部タブナビゲーションを作成する。
- 空状態・完了画面はキャラクター画像を使わず、文言とアイコンで状態を伝える。
- 保護者の契約状態は仮で `trial` として保存する。

完了条件:

- スマホ幅でLPが自然に表示される。
- 保護者が新規登録できる。
- 保護者がログイン・ログアウトできる。
- ログイン後に保護者ホームへ遷移する。
- 未ログイン時はLP、ログイン時はホームが表示される。

### Phase 2: 子ども管理

目的: 保護者が子どもを登録し、子ども用ログイン情報を発行できる状態を作る。

実装タスク:

- 子ども一覧画面を作成する。
- 子ども追加画面を作成する。
- 子ども詳細画面を作成する。
- 子どものニックネームを登録できるようにする。
- 子どものログインIDを自動または手入力で作成できるようにする。
- 子どもの初期パスワードを設定できるようにする。
- 子ども最大3人制限を実装する。
- 子どもごとの現在ポイントを表示する。
- 子ども追加後に初期科目を作成する。
- 子ども追加後に初期ポイントルールを作成する。

完了条件:

- 保護者が子どもを1人以上追加できる。
- 子ども一覧から詳細へ移動できる。
- 子どものログイン情報を保護者が確認できる。
- 4人目の追加はできない。

### Phase 3: 科目・ポイントルール

目的: 申請時に使う科目と、承認時に使うポイント計算の初期形を作る。

実装タスク:

- 子どもごとの科目一覧を作成する。
- 科目の追加・編集・削除を実装する。
- 削除済み科目は新規申請で選べないようにする。
- テスト100点満点ルールを作成する。
- テスト50点満点ルールを作成する。
- 成績5段階評価ルールを作成する。
- 成績A/B/C評価ルールを作成する。
- その他申請は承認時にポイント手入力とする。
- MVPではおすすめ設定を `normal` の初期値として自動作成する。
- 段階制を先に実装し、計算制は後回しにする。

完了条件:

- 子どもごとに科目を管理できる。
- テスト・成績の基本ポイントが自動算出できる。
- 保護者が承認時に確定ポイントを修正できる。

### Phase 4: 子どもの申請体験

目的: 子どもがスマホから学習成果を申請できる状態を作る。

実装タスク:

- 子どもログインを実装する。
- 子どもホームを作成する。
- ポイント残高カードを表示する。
- 申請作成フローを作成する。
- 分類選択画面を作成する。
- テスト申請入力を作成する。
- 成績申請入力を作成する。
- その他申請入力を作成する。
- 写真アップロードを実装する。
- 申請確認画面を作成する。
- 申請完了画面を作成する。
- 子どもの申請履歴を作成する。
- 差し戻しコメントを子どもが確認できるようにする。

完了条件:

- 子どもがテスト・成績・その他の申請を作成できる。
- 申請には1枚以上の写真を添付できる。
- 申請後、ステータスが `pending` になる。
- 子どもが自分の申請履歴を確認できる。

### Phase 5: 保護者の承認・ポイント付与

目的: 保護者が申請を確認し、ポイント付与まで完了できる状態を作る。

実装タスク:

- 保護者ホームに未確認申請数を表示する。
- 保護者の申請一覧画面を作成する。
- 申請詳細画面を作成する。
- 申請写真を確認できるようにする。
- 分類・科目・点数・評価・ポイントを修正できるようにする。
- 承認を実装する。
- 修正して承認を実装する。
- 差し戻しを実装する。
- 却下を実装する。
- 承認時にポイント履歴を作成する。
- 承認時に子どもの現在ポイントを加算する。
- 承認時の確定ポイントを申請に保存し、以後のポイントルール変更では再計算しない。
- 子ども向け通知を作成する。
- 保護者向け通知の既読化を実装する。

完了条件:

- 保護者が申請内容と写真を見て判断できる。
- 承認すると子どものポイントが増える。
- 承認済み申請のポイントは、後からポイントルールを変更しても変わらない。
- 差し戻し・却下の結果を子どもが確認できる。
- ポイント増減履歴が保存される。

### Phase 6: 運営管理画面

目的: 親子向けスマホアプリとは別に、運営がPCでユーザー・契約・申請状況を確認できる状態を作る。

実装タスク:

- `/admin` 配下にPC向け管理画面を作成する。
- 運営者ログインを作成する。
- 保護者一覧を作成する。
- 保護者詳細を作成する。
- 子ども一覧を作成する。
- 子ども詳細を作成する。
- 申請一覧を作成する。
- 申請詳細を作成する。
- 契約状態・トライアル状態を確認できるようにする。
- 支払い状態を確認できるようにする。
- 問い合わせ一覧を作成する。
- 退会・利用停止状態を確認できるようにする。

完了条件:

- 運営がPCで保護者・子ども・申請の状況を横断的に確認できる。
- 親子向けスマホ画面と運営管理画面の導線が分離されている。
- 管理画面はPC幅を前提に、一覧・検索・絞り込みがしやすい構成になっている。

## 4. スマホファーストの画面構成

### 共通方針

- 画面幅はスマホを基準にする。
- PCでは中央にスマホ幅のコンテンツを表示する。
- 1画面1目的を基本にする。
- 下部タブナビゲーションを使う。
- 主要ボタンは親指で押しやすい大きさにする。
- 入力フローはステップ式にする。
- 一覧はカード表示にする。
- 重要操作は確認画面または確認ダイアログを挟む。
- 横スクロールは使わない。
- 色は明るいオレンジ系を主にし、背景は白または薄い暖色にする。
- キャラクター画像は使用しない。余白、色、アイコン、文言で親しみやすさを出す。

### 未ログイン

- LP
  - ファーストビュー

  - 使い方
  - 主な機能

8. キャラクター画像を使わない画面ルールを反映する。

### Phase 1開始前に決めること

- フロントエンドの技術スタック
- バックエンドの技術スタック
- 認証方式
- 画像保存先
- キャラクター画像は使用しない
- デプロイ先

技術選定が未確定の場合は、MVP速度を優先して以下を推奨する。

- Next.js
- TypeScript
- Tailwind CSS
- Supabase Auth
- Supabase Database
- Supabase Storage
- Vercel

この構成なら、Phase 1からPhase 5までを小さく作りながら本番に近い形で検証しやすい。
