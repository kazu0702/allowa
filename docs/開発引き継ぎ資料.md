# INCE 開発引き継ぎ資料

作成日: 2026年5月30日

## 1. 現在の状態

INCEは、親子で学習成果を申請・承認し、ポイントやおこづかい申請につなげるスマホファーストMVPとして開発中。

公開URL:

- https://ince-eight.vercel.app

GitHub:

- https://github.com/kazu0702/ince

現在はVercel Hobbyプラン、Supabase Freeプランで運用している。

## 2. 開発の出発点

最初に `/docs/要件定義.md` を読み、MVPの中心を以下に絞った。

- ほぼ100%スマホ利用前提
- トップページはLP
- キャラクター画像は使用しない
- 親子の申請・承認・ポイント付与体験をMVPの中心にする
- 課金は重要だが、初期の動作検証では後半に回す

この方針に基づき、`/docs/MVP開発計画.md` を作成し、Phase 1〜Phase 5の実装タスク、スマホファーストの画面構成、データ設計、開発順序を整理した。

## 3. 時系列の実装内容

### Phase 1: LPと基本導線

最初にスマホ向けLPを実装した。

理由:

- MVPの入口として、サービスの価値を短時間で伝える必要があったため
- 親子どちらもスマホで見る前提だったため

実装内容:

- LP
- キャラクター画像なしの実用寄りUI
- 親ログイン、子どもログイン導線
- 「子どものモチベーションが上がる理由」をLPに追加

### Phase 2: 保護者側の子ども管理

保護者が子どもを登録・確認できる画面を実装した。

理由:

- 親子体験の起点が「子どもを登録する」ことだったため

実装内容:

- 子ども一覧
- 子ども追加
- ログインID、パスワードの自動生成
- ログインIDとパスワードは編集不可に変更
- 子ども削除機能を追加
- ホームの「子ども一覧を見る」導線を修正

### Phase 3: ポイントルール

科目ごとにポイントルールを設定できるようにした。

理由:

- INCEの中核は「成績や学習成果をポイントに変換する体験」のため
- 後からルールを変えても、過去に承認済みのポイントが変動してはいけないため

実装内容:

- 科目管理
- テスト点数に応じたポイント設定
- 成績評価に応じたポイント設定
- 承認時のポイントを `fixedPoints` として保存
- 過去承認分がルール変更で変動しないように修正
- 段階評価の評価項目を文字列として扱う方式に変更
- 100点満点、50点満点ルールの固定行・未満条件を調整

### Phase 4: 子ども側の申請体験

子どもが自分で成果を申請できるようにした。

理由:

- 子どもが「自分で申請する」体験をMVPの中心に置いたため

実装内容:

- テスト申請
- 成績申請
- その他申請
- 写真アップロード
- その他の場合は写真必須を解除
- 申請履歴
- 確認待ち申請の修正・キャンセル
- キャンセル後の再申請
- キャンセル済み申請の削除
- 申請履歴カードに写真サムネイル表示
- サムネイルタップで拡大表示

### Phase 5以降: 保護者承認、運営管理、おこづかい

保護者が申請を確認し、承認・差し戻し・却下できるようにした。

理由:

- MVPの中心体験が「申請 → 承認 → ポイント付与」だから

実装内容:

- 保護者側の申請一覧
- 申請詳細
- 承認、差し戻し、却下
- 承認取り消し
- ポイント履歴
- おこづかい申請
- 申請中ポイントの仮差し引き
- 完了時に確定差し引き
- 完了取り消し
- 今月もらった/支給したおこづかい合計
- 運営側管理画面をPC向けに追加

### デモ・ログイン改善

デモ確認をしやすくするため、親・子どもそれぞれにデモログインを追加した。

理由:

- 入力なしでMVPの主要体験を確認できるようにするため

実装内容:

- 親デモログイン
- 子どもデモログイン
- デモデータ自動作成
- デモガイド画面

### 事業計画

3ヵ年の事業計画と収支シミュレーションを作成した。

理由:

- MVPの検証後、事業化判断に必要な課金ユーザー数、売上、費用、利益の見通しを整理するため

成果物:

- `/docs/3ヵ年事業計画.md`
- `/docs/3ヵ年事業計画_A4印刷用.html`

## 5. Supabase対応

当初は全データをブラウザの `localStorage` に保存していた。

課題:

- 親と子が別端末で使うとデータ共有できない
- スマホ実機確認や将来の事業化にはクラウド保存が必要

FirebaseとSupabaseを比較し、INCEではSupabaseを選択した。

理由:

- 親、子ども、申請、承認、ポイント、おこづかい履歴など関係性のあるデータが中心
- 将来、運営管理画面や収支・利用状況集計と相性が良い
- PostgreSQLで後から本番用の正規化設計へ移行しやすい
- FreeプランでMVP検証を開始できる

Supabaseプロジェクト:

- Project name: `ince`
- Project URL: `https://mbfvtnywqpvedixcviej.supabase.co`
- Region: Tokyo
- Plan: Free

Vercel環境変数:

- `INCE_SUPABASE_URL`
- `INCE_SUPABASE_ANON_KEY`

注意:

- `INCE_SUPABASE_ANON_KEY` にはSupabaseのPublishable keyを設定している
- Secret keyはフロントエンドに入れてはいけない

## 6. Supabaseの現在のデータ設計

MVP検証のため、まずは全アカウントデータをJSONスナップショットとして保存する方式にした。

理由:

- 既存アプリが `localStorage` の親アカウントJSONを中心に動いていたため
- 画面や既存処理を大きく壊さず、クラウド同期を最短で検証するため
- 本番用テーブル設計の前に、親子別端末で同期できるかを確認するため

テーブル:

- `public.account_snapshots`

主なカラム:

- `email`
- `snapshot`
- `updated_at`
- `created_at`

SQL:

- `/supabase/schema.sql`

現在のRLS/権限:

- RLS有効
- MVP検証用に `anon` / `authenticated` へ読み書き許可
- 本番前には必ずSupabase Authを導入し、親アカウントごとのアクセス制御に変更する必要がある

## 7. アプリ側のSupabase実装

既存の大きな `app.js` を壊さないため、Supabase同期は後読み込みの `cloud.js` に分離した。

関連ファイル:

- `/app/index.html`
- `/app/cloud.js`
- `/app/config.example.js`
- `/scripts/build.mjs`
- `/supabase/schema.sql`

動き:

- Vercelビルド時に `config.js` を生成
- `config.js` にSupabase URLとPublishable keyを埋め込む
- `cloud.js` が `localStorage` の `ince_parent_account` を監視
- 変更があればSupabaseの `account_snapshots` にupsert
- 設定画面に「クラウド保存」カードを追加表示

公開アプリで確認済み:

- 保存方式: `Supabase`
- 同期状態: `同期済み`

## 9. 重要な注意点

### 1. 現在のSupabase設計はMVP検証用

`account_snapshots` にJSONを丸ごと保存している。

これは検証速度を優先した設計であり、本番設計ではない。

本番前に必要な対応:

- Supabase Auth導入
- 親ユーザーごとの認可
- children / applications / point_transactions / redemptions などの正規化
- Storageで写真管理
- 画像圧縮と保存期間ポリシー

### 2. RLSポリシーは緩い

現在はMVP検証用に `anon` から読み書きできる状態。

外部ユーザーに広げる前には必ず変更する。

### 3. 写真はまだ本格的なクラウド保存ではない

現状の写真は既存アプリのデータ構造に含まれる。

本番ではSupabase Storageへ分離するべき。

### 4. 課金は未接続

課金画面や事業計画はあるが、Stripe等の実課金は未接続。

### 5. GitHub更新時の注意

Codex環境ではローカル `.git` 作成や通常の `git push` が制限される場合があった。

そのため一部ファイルはGitHub UI/API経由で反映した。

## 10. 次にやるべきこと

優先度順:

1. スマホ実機で公開URLを確認
2. 親デモログイン、子どもデモログインを別端末で確認
3. Supabaseにスナップショットが保存されることをTable Editorで確認
4. Supabase Auth導入方針を決める
5. 本番用データ設計へ移行する
6. 写真をSupabase Storageに移す
7. 招待コード制またはクローズドβ導線を作る
8. Stripeなどの課金導線を接続する

## 11. 推奨する次の技術方針

短期:

- 現在のスナップショット同期で実機検証
- 10〜20家庭程度のテストに向けて操作性を改善
- 写真容量の制限を追加

中期:

- Supabase Auth
- 親子アカウント分離
- 正規化テーブル
- Storage
- 管理画面の実データ接続

長期:

- 課金
- 招待コード
- 利用分析
- CS対応用の運営管理画面
- セキュリティレビュー

## 12. 引き継ぎ時の一言まとめ

INCEは、スマホ前提の親子学習ポイントMVPとして、LP、親子ログイン、子ども管理、ポイントルール、申請、承認、おこづかい申請、運営管理の主要体験まで実装済み。

Vercelで公開済み、Supabase Freeプランとの同期も接続済み。

ただし、現在のSupabase同期はMVP検証用のJSONスナップショット方式であり、本番化前にはSupabase Authと正規化DB設計への移行が必須。
